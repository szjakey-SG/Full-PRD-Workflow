# 研发任务拆分模板

## 文件头

```markdown
# {Product Name} — 研发任务拆分

| 项目 | 详情 |
|------|------|
| 版本 | v1.0 |
| Sprint 数量 | {N} |
| 团队规模 | {N} 人（后端 / 前端 / QA / DevOps）|
| 目标上线日期 | {date} |
| 总工时预估 | {N} 人天 |
| 日期 | {date} |
```

## Epic > Story > Task 三级结构

```markdown
## Epic {N}: {Epic 名称}

**业务目标：** {这个 Epic 解决的业务问题}
**验收标准：** {Epic 完成的判定条件}
**优先级：** P0 / P1 / P2
**预估工时：** {N} 人天
**负责角色：** 后端 / 前端 / QA

---

### Story {N}.{M}: {Story 名称}

**用户故事：** 作为 {角色}，我希望 {功能}，以便 {价值}
**验收条件：** {具体可验收的条件}
**优先级：** P0 / P1 / P2

#### Task 列表

| Task ID | 描述 | 负责角色 | 预估工时 | 依赖 Task | 验收条件 | 风险标注 |
|---------|------|---------|---------|----------|---------|---------|
| T{N}.{M}.1 | 创建 subscription 表 DDL | 后端 | 0.5天 | 无 | DDL 经 DBA Review | — |
| T{N}.{M}.2 | 实现 POST /subscriptions 接口 | 后端 | 1天 | T{N}.{M}.1 | 接口测试通过 | — |
| T{N}.{M}.3 | 扣款逻辑 + 通道对接 | 后端 | 2天 | T{N}.{M}.2 | 支付成功率 > 95% in staging | ⚠️ 高风险 |
```

## 关键路径标注

```markdown
## 关键路径（Critical Path）

影响上线时间的任务链（任何延误将直接推迟上线）：

T1.1.1 → T1.2.1 → T1.2.3 → T2.1.1 → T2.1.2 → 上线

关键路径总工时：{N} 人天
缓冲时间建议：{N} 人天（约 {%} 缓冲）
```

## 高风险 Task 专项列表

```markdown
## 高风险 Task 清单（必须双人 Review）

| Task ID | 描述 | 风险类型 | 主 Review 人 | 协 Review 人 |
|---------|------|---------|-------------|-------------|
| T1.2.3 | 扣款逻辑 + 通道对接 | 资金安全 | 架构师 | 后端 Lead |
| T2.1.1 | 账务分录写入 Ledger | 数据一致性 | 架构师 | 财务 |
| T3.1.2 | 生产数据库迁移脚本 | 不可逆变更 | DBA | 架构师 |
```

| T1.1.1 | ... | 张三 | 0.5天 | TODO |

**Sprint 验收条件：**
- [ ] {可验收条件1}
- [ ] {可验收条件2}

**风险与依赖：**
- 依赖：{外部依赖项}

## 角色分工参考

| 角色 | 主要负责的 Task 类型 |
|------|-------------------|
| 后端研发 | 数据库、接口实现、业务逻辑、消息队列 |
| 前端研发 | 页面实现、表单校验、接口联调 |
| QA | 测试用例执行、回归测试、接口测试 |
| DevOps | 环境配置、部署脚本、监控告警 |
| 架构师 | 技术方案 Review、关键接口设计 Review |

## 完整性检查

- [ ] 每个 P0 功能的 Task 排入 Sprint 1
- [ ] 所有 Task 都有明确的负责角色
- [ ] 所有依赖关系已标注（不得有循环依赖）
- [ ] 高风险 Task 已在专项列表中登记
- [ ] 总工时与团队容量匹配（人数 × Sprint天数 × 70%有效系数）
- [ ] 关键路径已识别并标注
