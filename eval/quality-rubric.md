# LLM 可执行质量评分 Rubric

> 版本：2.0.0
> 本文件供 Claude 在 Phase 6 时自动执行评分使用。
> 评分规则：先引用 PRD 中的具体内容支撑判断，再给出 1-5 分。禁止直接给分。

---

## 评分执行指令

<scoring_instructions>
对每一个维度，按以下步骤执行：
1. 在已生成的文档中找到对应内容，引用原文（≥1处具体引用）
2. 对照评分标准，判断得分（1-5 整数）
3. 给出得分理由（1-2 句话，基于引用内容）
4. 如有不足，给出具体的改进项

禁止行为：
- 不得在未引用原文的情况下直接给分
- 不得给出 "总体不错" 等模糊评价
- 不得跳过任何维度
</scoring_instructions>

---

## 维度 1：业务完整性（权重 25%，最高 25 分）

**评分标准：**

| 分数 | 标准 |
|------|------|
| 5 | 覆盖全部角色（含边缘角色）、所有主流程和异常路径、所有状态转换、所有【待确认】项均有原因说明，无遗漏 |
| 4 | 覆盖主要角色和主流程，异常路径 >= 80%，【待确认】项 <= 3 个，无主流程缺失 |
| 3 | 主流程存在，但异常处理或角色权限有明显缺失（>= 2 处）|
| 2 | 核心业务对象定义不完整，或状态机仅覆盖正向路径 |
| 1 | 仅有功能列表，无业务规则、无状态机、无异常处理 |

**自动拦截：** 任何 P0 功能模块缺失业务规则 → 本维度强制 1 分，总分 BLOCKED。

**检查点：**
- [ ] 每个功能模块有业务规则章节
- [ ] 每个有状态的对象有完整状态机（Mermaid）
- [ ] 异常场景章节存在且非空
- [ ] 验收标准具体可测（非"功能正常"等模糊表述）
- [ ] 【待确认】项有原因说明

---

## 维度 2：工程可执行性（权重 25%，最高 25 分）

**评分标准：**

| 分数 | 标准 |
|------|------|
| 5 | 可直接拆分为 Task（有 Epic/Story/Task 文档）、可直接开发（有字段表和 API schema）、可直接测试（有用例 ID 和步骤）、可直接上线（有 Checklist）|
| 4 | 缺少 Task 拆分或 Checklist，但其他交付物完整 |
| 3 | PRD 和 API 存在，但测试用例覆盖率不足（< 70%的 API 有用例）|
| 2 | PRD 存在但 API 设计缺失，或测试用例为空 |
| 1 | 只有 PRD 草稿，无配套技术文档 |

**检查点：**
- [ ] API 端点数量 >= PRD 功能模块数 × 2（每个模块至少2个端点）
- [ ] 每个 API 端点有完整 Request + Response JSON schema
- [ ] 测试用例有 ID，步骤可操作
- [ ] Sprint Tasks 有明确负责角色和工时估算
- [ ] Release Checklist 有 Go/No-Go 判断标准

---

## 维度 3：系统一致性（权重 20%，最高 20 分）

**评分标准：**

| 分数 | 标准 |
|------|------|
| 5 | PRD 字段表 = ER 实体字段 = DDL 列名（完全一致），ER 关系 = DDL 外键，状态机状态 = domain/payment-states.md，无任何不一致 |
| 4 | 1-2 处 Minor 不一致（如字段注释不同），无 Critical/Major 不一致 |
| 3 | 存在 1-2 处 Major 不一致（字段名大小写、类型差异）|
| 2 | 存在 Critical 不一致（实体在 PRD 有，DDL 无，或反之）|
| 1 | 文档间高度不一致，无法作为工程依据 |

**自动拦截：** Completeness Report 中存在 Critical 问题且未修复 → 本维度强制 2 分。

**检查点：**
- [ ] 每个 PRD 实体在 ER 和 DDL 中都存在
- [ ] 字段名完全一致（snake_case，无大小写差异）
- [ ] 状态枚举与 domain/payment-states.md 一致
- [ ] ER 关系数 = DDL 外键数
- [ ] API 的状态转换与状态机一致

---

## 维度 4：风险控制覆盖（权重 15%，最高 15 分）

**评分标准：**

| 分数 | 标准 |
|------|------|
| 5 | 覆盖全部高风险场景（安全/权限/幂等/资金/合规），所有高风险 Task 标注 ⚠️，上线 Checklist 包含支付领域专项检查 |
| 4 | 覆盖大部分高风险场景，Checklist 完整，幂等设计存在 |
| 3 | 有基本的权限和幂等设计，但资金安全或合规检查缺失 |
| 2 | 仅有基本错误处理，无幂等策略，无支付领域专项检查 |
| 1 | 无任何风险控制相关内容 |

**支付领域专项检查点：**
- [ ] 金额字段精度有明确说明（DECIMAL位数和原因）
- [ ] 幂等键设计有描述（构成规则）
- [ ] 支付重试策略有说明（Dunning 或手动重试）
- [ ] 退款/拒付场景有独立测试用例
- [ ] 高风险转换（状态机）标注 ⚠️
- [ ] domain/security-rules.md 的安全声明已附加

---

## 维度 5：文档规范性（权重 15%，最高 15 分）

**评分标准：**

| 分数 | 标准 |
|------|------|
| 5 | 所有章节编号连续无跳级，字段表 4 列且无 bullet list 替代，Mermaid 图可渲染，文件头信息完整，【待确认】标注规范 |
| 4 | 1-2 处格式问题（如某章节编号不连续），不影响阅读 |
| 3 | 存在 bullet list 替代字段表，或 Mermaid 语法有错误 |
| 2 | 多处格式不规范，章节结构混乱 |
| 1 | 无结构化格式，纯自由文本 |

**检查点：**
- [ ] 所有字段定义使用 Markdown 表格（无 bullet list）
- [ ] 所有状态机使用 Mermaid stateDiagram-v2
- [ ] API 端点有 JSON 代码块（非文字描述）
- [ ] 文件头包含版本、日期、作者
- [ ] 【待确认】标注格式：【待确认: <原因>】

## 最终评分汇总模板

```markdown
## Quality Gate Report — {Project} v{version}

评分日期：{date}
评分模型：{model_name}

| 维度 | 权重 | 原始分(1-5) | 加权分 | 关键引用 | 判定 |
|------|------|-----------|--------|---------|------|
| 业务完整性 | 25% | {N} | {N×5} | "{原文引用}" | PASS/FAIL |
| 工程可执行性 | 25% | {N} | {N×5} | "{原文引用}" | PASS/FAIL |
| 系统一致性 | 20% | {N} | {N×4} | "{原文引用}" | PASS/FAIL |
| 风险控制覆盖 | 15% | {N} | {N×3} | "{原文引用}" | PASS/FAIL |
| 文档规范性 | 15% | {N} | {N×3} | "{原文引用}" | PASS/FAIL |
| **总分** | 100% | — | **{X}/100** | — | **{判定}** |

### 判定：{APPROVED / APPROVED WITH NOTES / NEEDS_REVISION / BLOCKED}

### 必须修复项（评分 < 80 时输出）
| # | 维度 | 问题描述 | 影响分数 | 修复建议 |
|---|------|---------|---------|---------|

### 改进建议（评分 >= 80 时输出）
- 建议1：{具体建议}
```

## 回归验证说明

本 rubric 同时用于 golden-cases/ 目录中的回归测试：
- 对每个 golden case 的期望输出执行评分
- 期望得分记录在 golden-cases/{case_name}/expected_score.json 中
- 实际评分与期望分差异 > 10 分时，触发人工 Review
